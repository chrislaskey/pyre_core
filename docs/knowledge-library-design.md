# Pyre Knowledge Library — Design Exploration

## Problem

Pyre's personas encode general conventions (Phoenix patterns, test strategies, review checklists) but have no mechanism for **project-specific knowledge** — the architecture decisions, data model, component library, coding style, and domain rules that make a real codebase unique. Without this context, each agent run starts from scratch, producing generic output that may not match the project's established patterns.

We want a **knowledge library** that:

1. Ships with sensible defaults, generated on `mix pyre.install`
2. Can be **regenerated by LLMs** (e.g. "scan this codebase and update the architecture doc")
3. Preserves **human input** across regenerations (corrections, preferences, domain context)
4. Is referenced by personas during the agent pipeline
5. Lives in the consuming project, not in the Pyre library

## How the Current System Works

For context on where knowledge fits into the pipeline:

- **System prompt** = persona markdown (`--append-system-prompt`)
- **User prompt** = feature request + prior stage artifacts + output instructions (`-p`)
- **Tool access** = run directory via `--add-dir`

Persona files are the only "knowledge" today. They live at `priv/pyre/personas/*.md` and encode role instructions + output format. There's no separate project context layer.

## Option A: Multi-File Per Topic (`*.prompt.md` + `*.human.md` + `*.md`)

Each knowledge topic gets three files in `priv/pyre/knowledge/`:

```
priv/pyre/knowledge/
  architecture.prompt.md      # LLM instructions for generating this doc
  architecture.human.md       # Human-authored notes, corrections, preferences
  architecture.md             # Generated result (LLM output merging prompt + human input)

  data_model.prompt.md
  data_model.human.md
  data_model.md

  ui_components.prompt.md
  ui_components.human.md
  ui_components.md

  code_style.prompt.md
  code_style.human.md
  code_style.md
```

**Workflow:**
1. `mix pyre.install` creates `*.prompt.md` files with default prompts
2. User optionally writes `*.human.md` with project-specific notes
3. `mix pyre.knowledge.generate` runs each `*.prompt.md` against the codebase, reads the corresponding `*.human.md`, and writes the `*.md` result
4. Personas reference the `*.md` files during pipeline runs

**Regeneration command:**
```bash
mix pyre.knowledge.generate              # regenerate all
mix pyre.knowledge.generate architecture # regenerate one topic
```

### Pros
- Clean separation of concerns: prompt vs human input vs output
- Human input never gets overwritten (it's in a separate file)
- Easy to understand what each file does by its extension
- `*.prompt.md` files are themselves versionable and improvable
- Can regenerate `*.md` without losing `*.human.md`
- Supports `git diff` on generated output to review changes

### Cons
- File proliferation: 3 files per topic (15+ files for 5 topics)
- Need to explain the convention to users
- Empty `*.human.md` files look odd when users haven't added anything
- The `*.prompt.md` -> `*.md` relationship is implicit (convention-based)

### Open Questions
- Should `*.human.md` be created empty on install or only when the user creates it?
- How do personas reference knowledge files — embed content in system prompt, or pass via `--add-dir`?
- Should `*.prompt.md` be gitignored (they're scaffolding) or committed (they're part of the project)?

## Option B: Two-File Per Topic (`*.prompt.md` + `*.md`)

Simpler variant — human input goes directly into the generated `*.md` file, marked with a special section:

```
priv/pyre/knowledge/
  architecture.prompt.md    # LLM instructions for generating
  architecture.md           # Generated result + human additions

  data_model.prompt.md
  data_model.md
```

**The `*.md` file structure:**
```markdown
# Architecture

<!-- GENERATED — will be refreshed by `mix pyre.knowledge.generate` -->

## Overview
Phoenix 1.8 application using LiveView for all interactive pages...

## Key Patterns
...

<!-- END GENERATED -->

## Human Notes

<!-- Everything below this line is preserved across regenerations -->

- We use a service layer between contexts for complex workflows
- All monetary values use the Money library, never raw floats
```

**Workflow:**
1. `mix pyre.install` creates `*.prompt.md` and initial `*.md`
2. User adds notes below the `<!-- Human Notes -->` marker
3. `mix pyre.knowledge.generate` replaces content between `GENERATED` markers, preserves everything after `Human Notes`

### Pros
- Fewer files (2 per topic instead of 3)
- Everything about a topic is in one readable document
- Human notes and generated content are in context together
- Simpler mental model for users

### Cons
- Marker-based parsing is fragile — users may accidentally edit/remove markers
- Merge conflicts are more likely since generated and human content share a file
- Harder to `git diff` just the generated portion
- The "don't edit above this line" convention is easy to violate
- Regeneration logic is more complex (parse markers, preserve sections)

### Open Questions
- What happens if a user removes the markers?
- Should the generated section be at the top or bottom?
- How to handle the case where the user wants to override a generated section?

## Option C: Single File with Frontmatter (`*.md` only)

One file per topic. YAML frontmatter holds the generation prompt. Body is the combined output:

```
priv/pyre/knowledge/
  architecture.md
  data_model.md
  ui_components.md
  code_style.md
```

**File structure:**
```markdown
---
prompt: |
  Analyze this Phoenix project's architecture. Document the
  directory structure, key modules, data flow patterns, and
  any architectural decisions visible in the code.
human: |
  - We use a service layer pattern for cross-context workflows
  - All API responses go through a normalizer module
regenerated_at: 2026-02-20T14:30:00Z
---

# Architecture

## Overview
Phoenix 1.8 application using LiveView...
```

**Workflow:**
1. `mix pyre.install` creates files with default `prompt:` frontmatter
2. User adds notes to the `human:` frontmatter field
3. `mix pyre.knowledge.generate` reads frontmatter, runs prompt + human notes against codebase, rewrites the body below the frontmatter

### Pros
- Minimal file count (1 per topic)
- Self-contained: prompt, human input, and output all in one file
- Frontmatter is a well-understood pattern (Jekyll, Hugo, etc.)
- Clean separation without markers in the body
- Body is pure readable markdown, no special syntax

### Cons
- Frontmatter YAML can be awkward for multi-line prompts and human notes
- Users unfamiliar with frontmatter may be confused
- The `human:` field in YAML is less inviting to write in than a markdown section
- Need a YAML parser (though Elixir has `:yaml_elixir`)
- Entire body gets overwritten on regeneration — user can't safely edit it
- Adds a dependency (yaml_elixir) or requires manual frontmatter parsing

### Open Questions
- Is a YAML dependency acceptable, or should we parse frontmatter manually?
- Is the `human:` field in YAML too constrained for longer human notes?

## Option D: Directory Per Topic

Each topic is a directory containing a fixed set of files:

```
priv/pyre/knowledge/
  architecture/
    prompt.md       # Generation instructions
    human.md        # Human notes (optional)
    output.md       # Generated result
  data_model/
    prompt.md
    human.md
    output.md
```

### Pros
- Very organized, each topic is self-contained
- Easy to add more files per topic later (e.g. `examples.md`, `changelog.md`)
- Clear which files are machine-generated vs human-authored
- `ls priv/pyre/knowledge/` shows topics at a glance

### Cons
- Deepest nesting — `priv/pyre/knowledge/architecture/output.md` is verbose
- Most filesystem overhead
- Feels over-engineered for what might be 4-5 topics
- Harder to scan all knowledge at once (need to open multiple dirs)
- Generic filenames (`output.md`) are less descriptive

### Open Questions
- Is the organizational benefit worth the nesting?
- Would users actually add more files per topic, or is this YAGNI?

## Cross-Cutting Design Questions

### 1. What knowledge topics should ship by default?

Candidates:
- **Architecture** — Project structure, key modules, data flow
- **Data model** — Schemas, relationships, important queries
- **UI components** — Available components, patterns, styling conventions
- **Code style** — Naming conventions, patterns, things to avoid
- **Domain/business rules** — Business logic, validation rules, edge cases

### 2. How do personas consume knowledge?

**Option 2a: Append to system prompt**
Concatenate relevant knowledge files after the persona markdown in `--append-system-prompt`. Simple, but makes the system prompt very long.

**Option 2b: Include in user prompt**
Add a `## Project Knowledge` section in the user prompt (in `build_prompt`), between the feature request and prior artifacts. Keeps system prompt focused on role instructions.

**Option 2c: Place in run directory**
Copy knowledge files into the run directory (or add the knowledge directory via another `--add-dir`). The persona instructions tell the agent to read them. Agents choose which files to reference based on relevance.

**Recommendation: 2b** — include in user prompt. It's the most direct, doesn't rely on the agent choosing to read files, and keeps the system prompt clean for role instructions.

### 3. Which stages get which knowledge?

Options:
- **All stages get all knowledge** — Simple, but increases prompt length for every stage
- **Stage-specific knowledge** — PM gets domain rules, Designer gets UI components, Programmer gets architecture + code style, etc.
- **Configurable per stage** — Define `knowledge: [...]` in each Stage struct

**Recommendation:** Start with all stages getting all knowledge. Optimize later if prompt length becomes an issue.

### 4. How does regeneration work?

The `mix pyre.knowledge.generate` command would:
1. Find all knowledge topics (by scanning the directory for `*.prompt.md` files)
2. For each topic, read the prompt file
3. Read the human notes file (if it exists)
4. Run a claude CLI call with the prompt + human notes, with access to the project codebase
5. Write the result to the output file

This is itself a single-agent call per topic — much simpler than the full pipeline.

### 5. Should knowledge files be gitignored?

- **Prompt files**: Committed (they're project config)
- **Human files**: Committed (they're human-authored)
- **Generated output**: Committed (they're part of project context, and regeneration is optional)

All knowledge files should be committed. They represent the project's accumulated understanding.

## Recommendation: Option A (Multi-File) with Simplifications

**Use Option A (`*.prompt.md` + `*.human.md` + `*.md`)** with these refinements:

1. **Don't create empty `*.human.md` files on install.** Only the `*.prompt.md` files are scaffolded. The `*.human.md` files are created by users when they have feedback. The generate command checks for their existence.

2. **Generate initial `*.md` files during install.** Run the generation step as part of `mix pyre.install` so users get populated knowledge files immediately. This means the install takes longer (LLM calls) but the result is more useful.

3. **Start with 4 topics:**
   - `architecture` — Project structure, modules, patterns
   - `data_model` — Schemas, relationships, queries
   - `ui_components` — Available components, styling patterns
   - `code_style` — Naming, patterns, anti-patterns

4. **Inject into user prompt** via a new `## Project Knowledge` section in `build_prompt`. Knowledge is loaded from the knowledge directory and included between the feature request and prior artifacts.

5. **File layout:**
   ```
   priv/pyre/knowledge/
     architecture.prompt.md
     architecture.md              # generated
     data_model.prompt.md
     data_model.md                # generated
     ui_components.prompt.md
     ui_components.md             # generated
     code_style.prompt.md
     code_style.md                # generated
     # created by users as needed:
     # architecture.human.md
     # data_model.human.md
   ```

### Why Option A over the others

- **vs Option B (two-file with markers):** Marker-based parsing is fragile and creates a "don't touch this" zone in a file users are supposed to own. Separate files are more honest about what's generated vs human-authored.

- **vs Option C (frontmatter):** YAML frontmatter works for short metadata but is awkward for multi-paragraph human notes and detailed prompts. Adds a parsing dependency. The body can't be safely edited by humans since it gets overwritten.

- **vs Option D (directories):** Over-engineered for 4-5 topics with 2-3 files each. Flat files with naming conventions are simpler and easier to scan.

### Why three files instead of two

The `*.human.md` file solves the core tension: **humans need to give feedback that survives regeneration.** Without it, either:
- The generated file can't be regenerated (because human edits would be lost)
- The generated file has fragile markers that users must respect

With a separate `*.human.md`, the contract is simple:
- Write your notes in `*.human.md` -> they always survive
- The `*.md` file can always be fully regenerated
- The generate command reads `*.human.md` and incorporates it into the output
